# 스티칭 알고리즘 정책, 선택지, 추천안 정리

작성일: 2026-03-06

## 1. 이 문서의 목적

이 문서는 현재 프로젝트에서 쓰고 있는 스티칭 정책과, 대화 과정에서 정리된 후보 정책들을 한데 모아 정리한다.

정리 대상은 아래 네 가지다.

- 특징점 매칭 정책
- 기하 변환 추정 정책
- 블렌딩 정책
- seam-cut 내부 정책

추가로 아래 내용도 포함한다.

- 딥러닝 매칭 계열 설명
- 현재 카메라 환경에 맞는 추천안
- 초기 보정 모드 / 런타임 모드 / 재보정 모드 정책
- 실시간에 추가할 수 있는 후보 기능 랭킹

## 2. 현재 카메라/프로젝트 환경 요약

현재 대화 기준으로 전제는 다음과 같다.

- 카메라 2대는 고정 rig다.
- 두 카메라는 서로 좌우로 떨어져 있고, 사람 눈처럼 약간 다른 방향을 본다.
- 즉 완전한 평행 배치는 아니고, 겹치는 영역이 있다.
- 가까운 전경보다 먼 장면을 주로 본다.

이 환경의 의미는 중요하다.

- 카메라가 고정이므로, 초기에 잘 맞춘 변환을 오래 재사용하기 좋다.
- 피사체가 멀수록 parallax 문제가 줄어든다.
- 따라서 "매 프레임 무거운 재보정"보다 "초기 1회 정밀 보정 + 런타임 고정 사용"이 더 잘 맞는다.

## 3. 현재 프로젝트에서 실제로 선택된 정책

현재 코드 기준으로는 다음이 핵심이다.

### 특징점 매칭

- 기본: `ORB + BFMatcher + ratio test`
- 데스크톱 경로에서는 가능하면 `CUDA ORB matcher` 시도 후 실패 시 CPU ORB 폴백

관련 파일:

- `stitching/core/features.py`
- `stitching/desktop_app.py`

### 기하 변환 추정

- 기본: `Homography + RANSAC`
- fallback: `Affine Partial 2D`

관련 파일:

- `stitching/core/geometry.py`

### 블렌딩

- `feather`
- `seam_cut`

관련 파일:

- `stitching/core/blend.py`

### seam-cut 내부

- cost map 계산
- seam path 찾기
- seam 기반 transition blend

현재 cost는 대략:

- 밝기 차이
- gradient 크기
- 옵션에 따라 temporal motion cost

## 4. 특징점 매칭 정책

가장 자주 쓰는 것부터 정리한다.

### 4-1. ORB + BFMatcher + ratio test

현재 프로젝트 기본값이다.

장점:

- 빠르다
- OpenCV 기본 지원이 좋아서 붙이기 쉽다
- binary descriptor라 매칭 비용이 비교적 낮다
- 실시간 지향 시스템에서 가장 현실적이다

단점:

- 반복 패턴에 약할 수 있다
- 질감이 적은 장면에서 흔들릴 수 있다
- scale/시점 변화에 더 강한 계열보다 품질이 밀릴 수 있다

특징:

- "속도 우선 기본기"에 가깝다
- 지금 프로젝트가 이걸 쓰는 이유도 실시간성 때문이다

### 4-2. CUDA ORB + GPU matcher

현재 데스크톱 경로에서 가능하면 시도하는 방식이다.

장점:

- ORB 구조를 유지하면서 초기 보정/재보정 속도를 올릴 수 있다
- CPU ORB보다 일부 환경에서 유리할 수 있다

단점:

- 환경 의존성이 있다
- OpenCV CUDA 지원 상태에 따라 실패할 수 있다
- 항상 CPU ORB보다 잘 되는 건 아니다

특징:

- "ORB를 GPU로 더 밀어붙인" 형태지, 완전히 다른 정책은 아니다

### 4-3. AKAZE

현재 프로젝트엔 직접 들어있지 않지만 전통적인 대안이다.

장점:

- 경계/윤곽 기반 장면에서 강한 편이다
- binary 계열이라 SIFT보다 가볍다

단점:

- ORB보다 느린 편이다
- GPU 경로가 ORB만큼 실용적이지 않다

특징:

- ORB가 계속 흔들릴 때 중간 대안으로 검토할 수 있다

### 4-4. SIFT

초기 1회 보정용으로 가장 현실적인 강한 후보였다.

장점:

- ORB보다 안정적인 경우가 많다
- scale/rotation 변화에 강한 편이다
- 초기 정밀 정합에 유리하다

단점:

- 느리다
- descriptor가 커서 매칭 비용이 크다
- 매 프레임 실시간용으론 불리하다

특징:

- "초기 1회 정밀 보정"에는 가치가 크다
- "런타임 실시간 반복"에는 불리하다

### 4-5. 딥러닝 매칭

주로 거론된 후보는 아래였다.

#### SuperPoint

- 특징점과 descriptor를 신경망으로 만든다

장점:

- 저질감/조명 변화/약간의 blur에서 ORB보다 안정적일 수 있다

단점:

- 추론 비용이 있다

#### LightGlue

- 특징점을 받아 더 똑똑하게 매칭한다

장점:

- 좋은 후보 match를 고르는 데 강하다

단점:

- 역시 추론 비용이 있다

특징:

- `SuperPoint + LightGlue` 조합으로 많이 본다

#### LoFTR

- sparse keypoint 기반보다 더 dense한 대응점을 찾는 계열이다

장점:

- 질감이 적은 장면, 반복 패턴에서 강한 경우가 있다

단점:

- 더 무겁다
- 실시간 목표엔 특히 불리하다

특징:

- "품질 최우선 초기 보정" 정도에서만 검토 가치가 있다

### 4-6. 딥러닝 매칭에 대한 현재 판단

현재 프로젝트 기준 추천은 이렇다.

- 런타임 매 프레임 딥러닝 매칭: 비추천
- 초기 1회 보정용 딥러닝 매칭: 검토 가치 있음
- 재보정 전용 딥러닝 매칭: 필요 시 후보

즉 딥러닝은 "실시간 hot path"보다 "초기 보정 또는 드문 재보정" 쪽에 더 맞는다.

## 5. 기하 변환 추정 정책

### 5-1. Homography + RANSAC

현재 메인 정책이다.

장점:

- 파노라마 스티칭의 가장 일반적인 기본 해법이다
- 평면 장면, 멀리 있는 배경, 고정 카메라 rig에 잘 맞는다
- outlier를 RANSAC으로 어느 정도 걸러낼 수 있다

단점:

- 강한 parallax가 있으면 완벽히 설명하지 못한다
- 가까운 움직이는 물체가 있으면 고스팅과 왜곡이 생기기 쉽다

특징:

- 지금 환경처럼 "멀리 있는 장면 위주"에선 가장 현실적이다

### 5-2. Affine Partial 2D fallback

현재 프로젝트에 이미 보조 정책으로 들어 있다.

장점:

- homography가 너무 불안정할 때 더 보수적으로 맞출 수 있다
- 회전/이동 중심 장면에선 오히려 더 안정적일 수 있다

단점:

- 표현력이 약하다
- 원근 차이까지는 잘 못 따라간다

특징:

- "메인 정책"이 아니라 "안전장치"로 맞다

### 5-3. Full Affine

대화 중 질문이 나왔던 정책이다.

장점:

- partial affine보다 더 자유도가 있다
- 카메라 배치가 단순하고 고정돼 있으면 충분할 수 있다

단점:

- Homography보다 표현력이 약하다
- toe-in 배치와 원근 차이가 존재하면 부족할 수 있다

특징:

- "카메라가 거의 같은 방향을 보고, 좌우로만 나란히 떨어진" 단순 rig에선 쓸 수 있다
- 지금 프로젝트의 메인 기본값으로는 Homography가 더 맞는다

### 5-4. Translation / Euclidean / Similarity

이건 더 단순한 모델들이다.

장점:

- 빠르다
- 잘못 맞을 자유도가 적다

단점:

- 표현력이 너무 약하다
- 카메라가 거의 완전히 정렬된 경우가 아니면 잘 못 맞는다

여기서 "정렬된 상황"이란:

- 두 카메라가 거의 같은 방향을 보고
- 높이 차이가 거의 없고
- 스케일 차이도 거의 없고
- 사실상 옆으로 조금 밀린 정도인 경우를 말한다

특징:

- 현재 프로젝트의 카메라 배치에는 메인 정책으로 부적합하다

### 5-5. Local warp / Mesh warp / APAP

고급 품질 쪽 대안이다.

장점:

- parallax 문제를 전역 homography보다 더 잘 다룰 수 있다

단점:

- 계산이 무겁다
- 구현이 복잡하다
- 실시간 목표와 충돌하기 쉽다

특징:

- 품질은 좋아질 수 있지만 현재 목표 fps에는 불리하다

## 6. 블렌딩 정책

### 6-1. Feather blending

현재 프로젝트의 기본 블렌딩이다.

장점:

- 빠르다
- 구현이 단순하다
- GPU로 옮기기 쉽다
- 정적인 장면에서는 자연스러울 수 있다

단점:

- 겹치는 영역에 움직이는 물체가 있으면 고스팅이 심해진다
- 좌우 차이가 크면 두 장이 겹쳐 보이는 느낌이 남는다

특징:

- "속도 우선" 정책이다

### 6-2. Seam-cut blending

현재 프로젝트에서 overlap 차이가 클 때 쓰는 정책이다.

장점:

- feather보다 고스팅을 줄일 수 있다
- "어디를 기준으로 자를지"를 계산하기 때문에 더 똑똑하다

단점:

- 계산이 더 무겁다
- seam이 이상하면 경계가 보일 수 있다

특징:

- "품질 우선" 쪽 정책이다
- 현재 프로젝트에서는 매우 중요하지만 실시간성엔 부담이 있다

### 6-3. Multi-band blending

대화 중 설명한 품질 중심 대안이다.

장점:

- 큰 구조 차이와 색 차이를 더 자연스럽게 숨길 수 있다
- 정적 장면 품질은 좋을 수 있다

단점:

- feather보다 확실히 무겁다
- seam-cut보다도 대체로 무거운 축에 속한다
- 여러 해상도 피라미드와 임시 버퍼가 필요하다
- 현재 실시간 목표에는 불리하다

특징:

- 오프라인이나 매우 잘 최적화된 네이티브 런타임 쪽에 더 어울린다

### 6-4. Hard cut

장점:

- 매우 빠르다

단점:

- 경계가 노골적으로 보인다

특징:

- 디버그나 극단적 저비용 상황 외에는 잘 안 쓴다

## 7. seam-cut 내부 정책

seam-cut은 내부적으로 세 부분으로 나뉜다.

- cost map
- seam path 탐색
- seam 적용 방식

### 7-1. cost map 정책

#### 밝기 차이 기반

장점:

- 단순하고 빠르다

단점:

- 구조적 에지나 움직임을 반영하기 어렵다

#### 밝기 차이 + gradient

현재 프로젝트가 실질적으로 이 조합이다.

장점:

- 경계 구조를 조금 더 반영할 수 있다
- seam이 티 나는 위치를 피하기 쉬워진다

단점:

- 복잡한 움직임엔 한계가 있다

#### motion-aware cost

현재 코드에 개념적 흔적이 있는 정책이다.

장점:

- 움직이는 전경 위로 seam이 지나가는 걸 줄일 수 있다

단점:

- 이전 프레임 의존성이 생긴다
- 계산량이 증가한다

#### semantic/object-aware cost

장점:

- 사람/차량 같은 중요한 물체를 피하는 seam이 가능해진다

단점:

- 딥러닝 segmentation 등 무거운 단계가 필요하다

특징:

- 장기 후보이지 현재 우선순위는 낮다

### 7-2. seam path 탐색 정책

#### Dynamic Programming

현재 프로젝트가 쓰는 방식이다.

장점:

- 구현이 단순하다
- row-wise seam 문제에 현실적이다

단점:

- 더 강한 전역 최적화 방식보다 한계가 있다

#### Graph Cut

장점:

- 복잡한 경우에 더 좋은 seam을 찾을 수 있다

단점:

- 더 무겁고 구현이 복잡하다

#### Fixed seam

대화에서 런타임 최적화용으로 권장된 방향이다.

장점:

- 매우 빠르다
- 초기 1회 계산 후 런타임 재사용이 가능하다

단점:

- 장면 변화가 크면 seam이 안 맞을 수 있다

특징:

- 지금 카메라 환경에 매우 잘 맞는다

### 7-3. seam 적용 정책

#### seam 주변 선형 transition

현재 프로젝트가 쓰는 방식이다.

장점:

- hard cut보다 자연스럽다
- 계산 비용이 합리적이다

단점:

- transition 폭이 너무 넓으면 ghost가 남을 수 있다

#### hard cut on seam

장점:

- 가장 빠르다

단점:

- 경계가 거칠게 보일 수 있다

#### multi-band around seam

장점:

- 품질은 더 좋아질 수 있다

단점:

- 다시 무거워진다

## 8. 수동 포인트와 딥러닝을 초기 보정에 쓰는 추천안

대화에서 정리된 중요한 결론은 이거였다.

- 수동 포인트는 초기 1회 보정 품질을 높이는 데 가치가 크다
- 자동 매칭과 섞어 쓰는 게 가장 좋다

추천 조합:

1. 사용자가 겹치는 구간에서 6~10쌍 수동 점을 찍는다
2. 수동 점으로 1차 Homography를 만든다
3. 그 상태에서 자동 매칭을 추가한다
4. 수동 점 + 자동 매칭을 합쳐 최종 H를 다시 추정한다
5. seam도 이 시점에 고정 계산한다

자동 매칭 추천 우선순위:

- 1순위: `SIFT + RANSAC`
- 2순위: `SuperPoint + LightGlue`
- 3순위: 정말 필요할 때만 `LoFTR`

이유:

- 초기 1회 보정은 느려도 괜찮다
- 대신 가능한 한 정확해야 한다

## 9. 초기 보정 모드 / 런타임 모드 / 재보정 모드 추천안

### 초기 보정 모드

목표:

- 가장 정확한 1회 정렬

추천:

- 해상도: 원본 유지
- 수동 포인트: 6~10쌍
- 자동 매칭: `SIFT` 또는 필요 시 `SuperPoint + LightGlue`
- 변환: `Homography`
- 블렌딩 준비: seam 계산까지 수행
- 결과 저장: H, seam, overlap, exposure 기준

### 런타임 모드

목표:

- 최대 FPS

추천:

- 고정 H 재사용
- 매 프레임 재매칭 안 함
- 기본 블렌딩은 feather 또는 고정 seam
- seam-cut 재계산 안 함
- drift 감지 외에는 재보정 안 함

### 재보정 모드

목표:

- 틀어졌을 때만 복구

추천:

- 저주기 점검
- 필요 시만 자동 매칭 수행
- 성공했을 때만 H/seam 갱신
- 수동 버튼 재보정 지원

## 10. 실시간에 추가할 수 있는 후보 랭킹

효과 대비 비용 순서로 정리하면 다음과 같다.

1. 고정 H 재사용
2. 고정 seam 재사용
3. drift 감지 + 이벤트성 재보정
4. 미세 보정
5. 움직이는 물체 마스크
6. 블렌딩 정책 자동 선택 강화
7. 재보정 전용 딥러닝 매칭
8. semantic seam 보조
9. depth-aware warp/blend
10. 매 프레임 딥러닝 매칭

여기서 현재 환경에 가장 맞는 건 상위 1~4번이다.

## 11. 현재 환경에서의 최종 추천

현재 카메라와 장면 전제를 반영하면, 최종 추천은 이 조합이다.

- 초기 보정:
  - 수동 포인트 6~10쌍
  - `SIFT + Homography + 강한 RANSAC`
  - 필요 시 `SuperPoint + LightGlue`
  - 고정 seam 생성

- 런타임:
  - 고정 H
  - feather 또는 고정 seam
  - 재매칭 없음

- 재보정:
  - drift 감지 시만
  - 저주기 점검
  - 필요 시 강한 자동 매칭 사용

한 줄로 요약하면:

- 초기 1회는 강하게
- 런타임은 단순하게
- 재보정은 드물게

이게 현재 프로젝트와 카메라 환경에 가장 잘 맞는 방향이다.
