### 주요 실험 결론

1. UI만 없애는 것으로는 부족했다
- headless에서도 stitched FPS 상한이 낮았다

2. FFmpeg direct 입력 실험을 했다
- `opencv`
- `ffmpeg-cpu`
- `ffmpeg-cuda`

3. 하지만 `ffmpeg-cuda`가 곧바로 더 빠르지 않았다
- 이유:
  - decode 결과가 결국 `rawvideo -> Python bytes -> numpy.ndarray`
  - 즉 Python-visible frame 경계를 벗어나지 못함

4. 따라서 병목의 본질은
- 특정 함수 하나가 아니라
- **Python이 full-rate frame을 소유하는 구조**라는 결론에 도달

## 2026-03-06 이후 설계 정리

### 의사결정

- Python은 control/UI/log 중심으로 남긴다
- hot path는 native runtime으로 이동한다

즉 목표 구조는:

```text
입력/디코드 -> pair/sync -> stitch -> encode -> output stream
```

그리고 이 media path는 native runtime이 소유한다.

### 구조 선택지 정리

- `A`: standalone native runtime
- `B`: GStreamer plugin/runtime
- `C`: FFmpeg 중심 runtime/filter

현재 판단:
- 먼저 `A`
- 이후 필요하면 `B` 또는 `C`

### 결과 확인 방식 정리

- Python이 full-rate stitched raw frame을 다시 받는 구조는 피한다
- 대신 final output stream을 외부 클라이언트처럼 본다

즉:
- native runtime이 최종 스트림을 전송
- Python은 viewer를 띄워 그 스트림을 구독
