# Python 런타임에서 네이티브 엔진으로 가는 의사결정 로그

작성일: 2026-03-06

## 1. 이 문서의 목적

이 문서는 현재 프로젝트가 왜 Python 중심 실시간 스티칭 프로토타입에서 출발했고, 왜 점점 "Python은 겉부분, 진짜 엔진은 네이티브 런타임" 구조로 방향을 바꾸게 되었는지를 정리한다.

핵심 질문은 하나였다.

- 어떻게 하면 두 개의 RTSP 영상을 실시간으로 스티칭해서, 장기적으로는 30fps를 넘고 최종적으로는 60fps 이상까지 노릴 수 있나?

이 문서는 그 질문에 답하기 위해 실제로 어떤 실험을 했고, 왜 어떤 선택을 했는지 순서대로 기록한다.

## 2. 출발점: Python/OpenCV 중심 구조

처음 구조는 Python/OpenCV 중심이었다.

- RTSP 입력을 OpenCV `VideoCapture`로 받는다.
- 좌우 프레임을 Python에서 버퍼링한다.
- Python에서 매칭, 정합, warp, blend를 제어한다.
- 필요 시 CUDA 경로를 사용한다.
- 결과를 데스크톱 창과 로그로 보여준다.

이 구조의 장점은 분명했다.

- 빠르게 만들 수 있다.
- UI, 수동 포인트 입력, 디버깅이 쉽다.
- 어떤 단계가 느린지 바로 로그로 볼 수 있다.
- 알고리즘을 실험하기 좋다.

즉 이 구조는 "제품"보다는 "참조 구현과 실험 플랫폼"으로는 매우 적합했다.

## 3. 첫 번째 문제: 스티칭된 화면 FPS가 낮다

처음 발견된 문제는 단순했다.

- 좌우 RTSP는 각각 30fps 근처로 들어온다.
- 그런데 stitched 화면 FPS는 그보다 훨씬 낮다.
- GPU를 켜도 기대만큼 올라가지 않는다.
- 고스팅도 심하다.

그래서 처음엔 다음 가설들을 차례대로 확인했다.

- CPU 병목인가?
- GPU를 덜 쓰는 건가?
- 화면 출력이 느린 건가?
- RTSP 입력이 느린 건가?
- Python 자체가 병목인가?

## 4. 첫 번째 해결책: 기존 Python/OpenCV 구조 안에서 최대한 최적화

처음에는 구조를 바꾸기보다, 현재 구조 안에서 어디까지 갈 수 있는지를 먼저 확인했다.

실제로 진행한 방향은 아래와 같았다.

- warp, blend의 GPU 사용 확대
- 불필요한 `copy()`와 메모리 재할당 제거
- overlap ROI만 계산하도록 축소
- 대시보드와 파노라마 창 갱신 주기 분리
- headless benchmark 추가

이 단계의 목적은 단순했다.

- "지금 구조 자체가 정말 안 되는 건지, 아니면 아직 최적화가 덜 된 건지"를 먼저 확인하는 것

## 5. headless benchmark가 알려준 사실

headless benchmark는 화면을 띄우지 않고, 순수 계산 속도만 보는 모드다.

이 실험이 중요했던 이유는 다음과 같다.

- 만약 UI가 문제라면, 화면만 끄면 FPS가 크게 올라야 한다.
- 그런데 화면을 꺼도 상한이 대략 20fps 안팎에 머물렀다.

이 결과는 중요한 결론을 줬다.

- 문제는 UI만이 아니다.
- 현재 Python/OpenCV 런타임 구조 자체가 상한을 만들고 있다.

즉 이 시점부터는 "조금 더 튜닝해보자"가 아니라 "구조를 다시 봐야 한다"가 되었다.

## 6. 두 번째 해결책: OpenCV 뒤에 숨은 FFmpeg 대신 FFmpeg를 직접 써보자

다음 가설은 입력 경로였다.

기존 구조는 OpenCV `VideoCapture`가 내부적으로 FFmpeg를 쓰는 형태였다.
이 방식은 편하지만, 우리가 직접 제어할 수 있는 범위가 좁았다.

그래서 다음 실험을 했다.

- OpenCV 입력 경로
- FFmpeg 직접 제어 CPU decode
- FFmpeg 직접 제어 CUDA decode

목적은 이거였다.

- RTSP ingest를 직접 제어하면 성능이 더 나아지는지 본다.
- 저지연 옵션, 하드웨어 decode, transport를 더 명확히 통제해 본다.

그래서 프로젝트에 직접 FFmpeg reader/runtime scaffolding을 추가했다.

## 7. 왜 FFmpeg direct가 오히려 더 느리게 보였는가

이 질문은 실제로 여러 번 나왔다.

질문:

- 왜 headless benchmark가 더 빠르고, FFmpeg를 직접 쓴 쪽은 17fps 정도로 더 느리게 나왔나?

답은 "headless"와 "FFmpeg direct"가 같은 종류의 변화가 아니기 때문이다.

- headless는 화면 표시 비용만 제거한다.
- FFmpeg direct는 입력 경로를 바꾼다.

그런데 당시 구현한 FFmpeg direct 경로는 다음과 같았다.

- FFmpeg가 decode한다.
- 결과를 `rawvideo bgr24`로 stdout 파이프에 쓴다.
- Python이 그 큰 raw frame을 읽는다.
- Python이 `numpy` frame으로 만든다.
- 그걸 다시 GPU로 올려 stitch한다.

즉 FFmpeg를 직접 썼지만, 구조는 여전히 이랬다.

`decode -> Python raw frame -> GPU upload -> stitch`

여기서 중요한 건 "raw frame이 Python으로 올라온다"는 사실이다.

그래서 FFmpeg direct 구현은 이런 이유로 오히려 느려질 수 있었다.

- 별도 `ffmpeg` subprocess 비용
- stdout 파이프를 통한 대용량 rawvideo 전달 비용
- Python 바이트 읽기 비용
- `numpy` frame 생성 비용
- 그 후 다시 GPU로 올리는 비용

즉 "FFmpeg를 직접 쓴다"는 것만으로는 빨라지지 않는다.
핵심은 "큰 raw frame이 Python으로 올라오느냐"였다.

## 8. 세 번째 결론: 병목의 본질은 Python raw frame 경계다

이 시점에서 문제는 더 명확해졌다.

지금 구조의 핵심 문제는:

- 입력이 Python 프레임이다
- 버퍼가 Python 프레임이다
- sync가 Python 프레임 기준이다
- 결과도 Python에서 관리된다

즉 Python이 큰 프레임을 직접 받는 구조가 병목의 본질이었다.

이 결론이 중요한 이유는 다음과 같다.

- OpenCV를 계속 써도 이 병목은 남는다.
- FFmpeg를 직접 써도 Python raw frame 경계를 유지하면 병목은 남는다.
- GStreamer를 쓰더라도 Python `appsink`로 큰 프레임을 받으면 결국 같은 문제다.

그래서 실질적인 결론은 하나였다.

- Python이 큰 프레임을 직접 받지 않는 구조로 가야 한다.

## 9. 여기서 등장한 A / B / C 선택지

이때부터 선택지는 세 가지로 정리했다.

### A. 별도 네이티브 엔진 프로그램

예: `stitch_runtime.exe`

구조:

- Python은 UI, 제어, 로그만 담당
- 별도 네이티브 프로세스가 decode, stitch, encode를 담당

장점:

- 지금 Python 프로젝트를 가장 많이 재사용 가능
- UI/운영 툴은 살리고 hot path만 분리 가능
- 시작하기 가장 현실적

단점:

- 별도 엔진을 만들어야 함
- Python과 엔진 사이 통신 설계가 필요

### B. GStreamer 플러그인

구조:

- `source -> decoder -> stitch plugin -> encoder -> sink`

장점:

- 실시간 미디어 파이프라인에 매우 적합
- 분기, 저지연, 재연결, 다중 출력 같은 제품 기능에 강함
- 장기적인 제품 구조로 좋음

단점:

- 파이프라인 규칙, 버퍼 타입, 타임스탬프, GPU 메모리 경로를 다 맞춰야 해서 어렵다
- Windows보다 Linux에서 훨씬 다루기 쉽다

### C. FFmpeg 중심 런타임/필터

구조:

- FFmpeg/libav 쪽 decode/encode/streaming 흐름 안에 커스텀 스티칭 처리 삽입

장점:

- 코덱, 프로토콜, 인코딩/송출 제어에 강함
- RTSP/RTMP/SRT/file 처리 쪽은 FFmpeg가 매우 강함

단점:

- API가 낮은 레벨이라 구현/디버깅 난이도가 높다
- 중간에 커스텀 스티칭 로직을 넣는 것이 만만하지 않다

## 10. 왜 지금은 A를 먼저 택했는가

여기서 중요한 선택이 있었다.

- 당장 목표는 "모든 걸 최종 제품 구조로 바꾸기"가 아니다.
- 현재 Python 프로젝트를 버리지 않고, 실험 자산을 유지한 채 hot path를 뽑아내는 것이다.

그래서 지금 시점의 합리적인 선택은 A였다.

선택 이유:

- Python UI, 수동 포인트, 로그, 모니터링은 이미 잘 갖춰져 있다.
- 이걸 다 버리고 GStreamer나 FFmpeg 필터로 바로 점프하는 건 리스크가 크다.
- A는 가장 적은 충돌로 성능 병목을 바깥으로 분리할 수 있다.

즉 현재 판단은:

- 지금은 A가 맞다
- 다만 A를 나중에 B/C로 갈 수 있게 설계해야 한다

## 11. A로 갔다가 나중에 B/C로 가면 또 다시 짜야 하는가

이 질문도 핵심이었다.

답은 "A를 어떻게 설계하느냐에 따라 다르다"였다.

잘못된 A:

- 입력, 스티칭, 출력, IPC, 설정이 한 exe 안에 엉켜 있는 통짜 구조

이 경우:

- 나중에 B나 C로 갈 때 사실상 다시 짜야 한다

좋은 A:

- `core engine`과 `wrapper`를 분리한 구조

이 경우:

- core engine은 재사용
- wrapper만 갈아끼워서 A/B/C를 대응 가능

즉 중요한 건 A냐 B냐 C냐보다, 엔진 경계를 어떻게 나누느냐였다.

## 12. 왜 GStreamer를 나중 후보로 남겼는가

이 프로젝트 목표가 장기적으로는 `60fps 이상 실시간 stitching + 송출`이기 때문에, GStreamer 얘기가 다시 나왔다.

왜 B가 장기적으로 강한가:

- 입력, decode, custom block, encode, output을 한 파이프라인에서 관리하기 좋다
- 운영 제품화에 유리하다
- 다중 출력, 낮은 지연, 재연결, queue 정책 같은 실무 기능이 좋다

왜 지금 바로 B로 안 가는가:

- 현재 저장소와 연결되는 경로가 가장 짧은 건 A다
- GStreamer는 최종적으로 강하지만 첫 점프 비용이 크다

즉 장기적으로는 B가 될 수도 있지만, 지금 단계에서 바로 B를 시작하는 건 비효율적일 수 있다는 판단이었다.

## 13. 왜 C도 유력 후보로 남겨두는가

만약 최종 요구가 송출 중심으로 강해지면, FFmpeg 중심 구조도 강해진다.

예:

- 다양한 코덱
- RTSP/RTMP/SRT 대응
- 하드웨어 encode/decode 관리

이런 요구는 FFmpeg가 매우 강하다.

그래서 현재 결론은:

- 1차는 A
- 이후 성능/운영 요구에 따라 B 또는 C 검토

즉 "A 다음은 반드시 B"가 아니라,
"A 이후 요구에 따라 B 또는 C"다.

## 14. 자주 나온 질문과 답변

### Q. 결국 이건 프로젝트를 처음부터 다시 만드는 거 아닌가?

A.

- 런타임 구조는 거의 새로 만드는 게 맞다.
- 하지만 알고리즘, 파라미터 의미, 수동 calibration UX, 로그/메트릭 구조, 실험 결과는 그대로 자산이다.

즉 완전한 백지 상태는 아니고, "핵심 실행 구조를 새로 짓는 것"에 가깝다.

### Q. VS Code에서 할 수 있나?

A.

- 가능하다.
- 다만 VS Code는 편집기일 뿐이고, 실제 빌드에는 Visual Studio 2022 툴체인과 CUDA Toolkit이 필요하다.
- 즉 "VS Code로 개발"은 가능하지만, 컴파일러는 바깥 툴체인을 쓴다.

### Q. Python에서 CUDA 쓰려고 고생한 건 다 버려지나?

A.

- 아니다.
- NVIDIA 드라이버, CUDA Toolkit, VS 2022 빌드 환경은 그대로 자산이다.
- Python용 OpenCV wheel 빌드는 C++ 엔진에 그대로 쓰이지 않을 뿐이다.

### Q. 입력이 나중에 RTSP에서 다른 형식으로 바뀌면 또 다 갈아엎어야 하나?

A.

- 잘 분리하면 그렇지 않다.
- `Input Adapter`만 바꾸면 된다.
- 갈아엎게 되는 건 입력 형식이 바뀌어서가 아니라, 입력 코드와 엔진 코드가 붙어 있을 때다.

## 15. 현재 시점의 정리

지금까지의 판단을 한 줄로 줄이면 이렇다.

- Python/OpenCV 구조는 참조 구현과 알고리즘 실험에는 적합하다.
- 하지만 30fps 이상, 특히 60fps급 실시간 처리/송출 런타임으로는 구조적 한계가 보인다.
- 따라서 Python은 UI/제어/로그를 담당하고, 실제 hot path는 네이티브 엔진으로 분리하는 방향이 맞다.
- 그 첫 단계는 A가 가장 현실적이고, 이후 필요하면 B/GStreamer 또는 C/FFmpeg 중심 구조로 갈 수 있도록 core/wrapper 분리를 지켜야 한다.

## 16. 현재 의사결정

현재까지의 최종 의사결정은 다음과 같다.

1. Python 프로젝트는 즉시 버리지 않는다.
2. 현재 Python 코드는 참조 구현과 운영/제어/UI 자산으로 유지한다.
3. hot path는 네이티브 런타임으로 분리하는 방향을 채택한다.
4. 첫 단계는 A, 즉 별도 네이티브 엔진 프로그램을 목표로 한다.
5. 단, A를 통짜로 만들지 않고 core engine + wrapper 구조로 만든다.
6. 나중에 필요하면 B 또는 C로 wrapper를 바꾸되, core engine은 재사용하는 방향을 유지한다.
