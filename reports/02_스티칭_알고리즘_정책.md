# 스티칭 알고리즘 정책

## 목적

이 문서는 현재 프로젝트에서 사용 중인 스티칭 관련 정책과, 대안이 되는 정책들의 장단점, 그리고 현재 카메라 환경 기준 추천안을 정리한다.

현재 환경 전제:
- 카메라 2대는 고정
- 좌우로 벌어져 있고 서로 약간 다른 방향을 봄
- 겹치는 구간이 있음
- 가까운 전경보다 먼 장면 중심

이 전제는 매우 중요하다.
왜냐하면:
- 초기 1회 정밀 보정의 가치가 크고
- 런타임에 매 프레임 무거운 재보정이 꼭 필요하지 않을 가능성이 높기 때문이다.

## 현재 파이프라인을 나누면

스티칭은 대략 아래 단계로 나뉜다.

1. 특징점 매칭 정책
2. 기하 변환 추정 정책
3. 블렌딩 정책
4. seam-cut 내부 정책

각 단계는 역할이 다르다.

- 특징점 매칭: 두 영상의 대응점 찾기
- 기하 변환: 어떻게 겹칠지 계산
- 블렌딩: 겹친 뒤 어떻게 섞을지 결정
- seam-cut 내부: seam-cut을 쓴다면 seam을 어떻게 찾고 적용할지 결정

## 1. 특징점 매칭 정책

### 현재 기본: ORB + BFMatcher + ratio test

현재 프로젝트 기본 방향이다.

장점:
- 빠름
- OpenCV 기본 지원이 강함
- binary descriptor라 비용이 상대적으로 작음
- 실시간 지향 구조에서 현실적

단점:
- 반복 패턴에서 약할 수 있음
- 질감이 적은 장면에서 흔들릴 수 있음
- scale/시점 변화가 크면 SIFT류보다 약한 경우가 많음

특징:
- 지금 프로젝트처럼 속도가 매우 중요한 경우 가장 현실적인 선택

### CUDA ORB

가능하면 쓰고, 안 되면 CPU ORB로 폴백하는 후보였다.

장점:
- ORB의 속도 장점을 유지하면서 GPU 사용 가능
- 초기 자동 보정 속도를 높일 수 있음

단점:
- 환경 의존성이 큼
- OpenCV CUDA ORB 지원 상태에 따라 흔들릴 수 있음

특징:
- ORB 계열의 가속판에 가깝다

### AKAZE

중간 정도의 대안이다.

장점:
- 경계/윤곽 기반 장면에서 강한 경우가 있음
- binary descriptor 기반이라 SIFT보다는 가벼움

단점:
- ORB보다 느린 경우가 많음
- GPU 경로가 약함

특징:
- ORB와 SIFT 사이 중간 성격

### SIFT

초기 1회 보정용으로 가장 현실적인 품질 강화 후보다.

장점:
- scale/rotation 변화에 더 강함
- ORB보다 안정적인 match가 나오는 경우가 많음
- 초기 정밀 보정 품질을 높이기에 좋음

단점:
- 느림
- descriptor가 무거움
- full-rate 실시간 루프에는 부담

특징:
- 런타임 매 프레임용보다 초기 보정용에 적합

### 딥러닝 매칭: SuperPoint / LightGlue / LoFTR

#### SuperPoint

역할:
- 특징점 검출 + descriptor 생성

장점:
- ORB보다 안정적인 특징을 찾는 경우가 많음
- 저질감/blur/조명 변화에서 도움 가능

단점:
- 추론 비용이 있음

#### LightGlue

역할:
- descriptor 매칭을 더 똑똑하게 수행

장점:
- 잘못된 match를 줄이는 데 유리
- SuperPoint와 궁합이 좋음

단점:
- 추론 비용 추가

#### LoFTR

역할:
- sparse keypoint 기반보다 더 직접적으로 대응점을 찾는 방식

장점:
- 저질감/반복 패턴 장면에서 강할 수 있음

단점:
- 매우 무거움
- 실시간 메인 루프와는 잘 안 맞음

### 현재 추천

현재 환경 기준:
- 런타임 메인 루프: `ORB`
- 초기 1회 보정: `SIFT` 우선 검토
- 더 강한 품질이 꼭 필요할 때만 `SuperPoint + LightGlue`

즉 딥러닝은 현재 기준으로는 **초기 1회 보정 또는 재보정 전용 후보**다.

## 2. 기하 변환 추정 정책

### 현재 기본: Homography + RANSAC

장점:
- 파노라마 문제에서 가장 흔한 기본 해법
- 고정 카메라 + 멀리 있는 장면에서 잘 맞을 가능성이 큼
- outlier 제거를 같이 할 수 있음

단점:
- 가까운 전경이나 강한 parallax가 있으면 완벽하지 않음

특징:
- 현재 카메라 환경에서는 메인 선택으로 타당

### 현재 fallback: Affine Partial

장점:
- homography가 너무 흔들릴 때 더 보수적
- 이동/회전 위주 상황에서 안정적일 수 있음

단점:
- 표현력이 약함
- 원근 변화 설명이 부족함

특징:
- 메인보다는 안전장치

### Full Affine

장점:
- partial affine보다 자유도가 높음

단점:
- homography보다 표현력은 여전히 약한 경우가 많고
- toe-in 배치에서 충분하지 않을 수 있음

특징:
- 카메라가 거의 같은 방향이고 장면 관계가 단순할 때 후보

### Translation / Similarity / Euclidean

장점:
- 매우 빠름
- 단순함

단점:
- 카메라 배치가 거의 완전히 정렬되지 않으면 표현력이 부족

특징:
- 현재 카메라 배치엔 메인 정책으로는 부적합

### Local warp / Mesh warp / APAP

장점:
- parallax 문제를 더 잘 다룰 수 있음

단점:
- 계산량 큼
- 구현이 훨씬 복잡함
- 실시간 목표와 충돌 가능성이 큼

특징:
- 품질 중심 고급 대안

### 현재 추천

- 메인: `Homography`
- fallback: `Affine Partial`
- local warp 계열은 현재 목표 fps를 생각하면 보류

## 3. 블렌딩 정책

### Feather blending

장점:
- 빠름
- 단순함
- GPU 처리와 잘 맞음

단점:
- 움직이는 물체가 겹침 영역에 있으면 고스팅이 심해짐
- 좌우 차이가 크면 두 장이 섞여 보임

특징:
- 속도 우선

### Seam-cut blending

장점:
- 고스팅을 feather보다 줄이기 좋음
- “어디서 자를지”를 계산하므로 품질이 더 나을 수 있음

단점:
- 더 무거움
- seam 계산 비용이 있음

특징:
- 품질 우선

### Multi-band blending

장점:
- 정적 장면 품질이 매우 좋을 수 있음
- 밝기 차이와 중간 규모 구조 차이를 더 자연스럽게 숨길 수 있음

단점:
- 계산량 큼
- 메모리 사용량 큼
- 현재 프로젝트의 실시간 목표에는 매우 불리

특징:
- 오프라인 품질 지향에 가까움

### Hard cut

장점:
- 매우 빠름

단점:
- 경계가 눈에 띄기 쉬움

특징:
- 디버그나 극단적 단순화 상황 외에는 부적합

### 현재 추천

- 기본 런타임: `Feather` 또는 `고정 seam 기반 blend`
- 필요할 때만 `Seam-cut`
- `Multi-band`는 현재 목표에서는 비추천

## 4. Seam-cut 내부 정책

### cost map 계산

현재는 대체로:
- 픽셀 차이
- gradient 크기

를 조합하는 방향이 현실적이다.

장점:
- seam이 티 덜 나는 위치를 찾기 쉬움

단점:
- 움직이는 전경까지 완벽히 해결하지는 못함

### seam path 탐색

현재 가장 현실적인 방법:
- `Dynamic Programming`

장점:
- 구현 단순
- 실시간성에 상대적으로 유리

단점:
- graph cut보다 단순함

대안:
- `Graph Cut`
  - 더 강할 수 있지만 더 무거움

### seam 적용

현재 가장 현실적인 방향:
- seam 주변 transition 폭을 둔 부드러운 cut

장점:
- hard cut보다 자연스러움
- multi-band보다 훨씬 가벼움

단점:
- 전환 폭이 넓으면 다시 ghost가 늘 수 있음

## 초기 보정 / 런타임 / 재보정 추천안

### 초기 보정 모드

목표:
- 가장 정확한 1회 정렬

권장:
- 수동 포인트 6~10쌍
- `SIFT + Homography + 강한 RANSAC`
- 더 필요하면 `SuperPoint + LightGlue` 검토
- 고정 seam 생성

### 런타임 모드

목표:
- fps 최우선

권장:
- 매 프레임 재매칭 금지
- 저장된 고정 H 재사용
- 기본은 feather 또는 고정 seam
- seam 재계산 없음

### 재보정 모드

목표:
- 틀어졌을 때만 복구

권장:
- 저주기 점검
- drift 감지 시만 자동/수동 재보정
- 성공 시 새 H와 seam을 저장

## 수동 포인트는 어떻게 쓰는 게 맞나

수동 포인트는 두 가지 방식이 있다.

1. 수동 점만으로 H 계산
- 가능하지만 클릭 오차에 민감

2. 수동 점을 앵커로 쓰고 자동 매칭을 추가
- 더 좋음
- 수동 점으로 대략 정렬
- 자동 매칭으로 세부 정밀화
- 마지막에 RANSAC으로 최종 H 추정

현재 환경에서는 **2번이 더 좋다.**

## 실시간에 추가 가능한 후보들

효과 대비 비용 순서로 정리하면:

1. 고정 H 재사용
2. 고정 seam 재사용
3. drift 감지 + 이벤트성 재보정
4. 미세 보정(local micro-adjustment)
5. 움직이는 물체 마스크
6. 블렌딩 정책 자동 선택 강화
7. 딥러닝 매칭을 재보정 전용으로 사용
8. 딥러닝 기반 seam 회피
9. depth-aware warp/blend
10. 매 프레임 딥러닝 매칭

즉:
- 딥러닝은 “메인 루프”보다
- “초기 보정 / 재보정 / 보조 판단” 쪽이 현실적이다

## 최종 추천

현재 카메라 환경과 목표를 같이 보면:

1. 초기 1회 보정은 강하게
- 수동 포인트 + 강한 자동 매칭

2. 런타임은 단순하게
- 고정 H
- 고정 seam 또는 feather

3. 재보정은 드물게
- drift 감지 시만

즉 현재 가장 맞는 방향은:

> **초기 1회 강한 보정 + 런타임 고정 H/고정 seam + 필요 시만 재보정**

이다.
