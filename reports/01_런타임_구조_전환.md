# 런타임 구조 전환

## 목적

현재 프로젝트는 Python/OpenCV 기반 프로토타입으로 시작했고,

- RTSP 입력
- 자동/수동 보정
- 실시간 스티칭 미리보기
- 로그/모니터링

까지는 빠르게 실험할 수 있었다.

하지만 최종 목표는 다음이다.

```text
영상 입력(디코드) -> 스티칭 -> 영상 인코드 -> 네트워크 전송
```

그리고 이 목표는 단순 “기능 동작”이 아니라:

- 높은 stitched fps
- 낮은 지연
- 운영 중 안정성
- 나중에 송출 경로까지 포함한 확장성

을 요구한다.

이 문서는 왜 현재 구조를 유지하기 어렵다고 판단했는지, 어떤 실험을 했고 어떤 결론에 도달했는지, 앞으로 어떤 구조로 가는지가지를 정리한다.

## 현재 구조의 출발점

현재 Python 쪽 구조는 크게 보면:

```text
RTSP -> OpenCV/FFmpeg -> Python frame(np.ndarray) -> Python buffer/sync -> GPU upload -> stitch
```

즉:
- 입력 프레임이 Python에서 `numpy.ndarray`로 보이고
- Python이 프레임 버퍼와 pairing/sync를 소유하고
- GPU를 쓰더라도 stitch 직전까지는 Python 중심 흐름이다

이 구조는 프로토타입과 품질 검증에는 적합하다.

장점:
- UI/로그/실험 속도 빠름
- 수동 포인트 기능 붙이기 쉬움
- 알고리즘 실험 반복이 쉬움

단점:
- full-rate 프레임 hot path가 Python 객체 경계를 지나감
- host/device 업로드 경계가 자주 생김
- 나중에 encode/send까지 포함하면 구조적 부담이 커짐

## 무엇이 실제 병목이었나

가장 중요한 관찰은 이거였다.

1. 좌/우 입력 FPS는 30 근처로 들어옴
2. stitched FPS는 그보다 훨씬 낮았음
3. GPU를 켜도 기대만큼 올라가지 않았음

그래서 우리는 다음 순서로 확인했다.

### 1단계. 기존 Python/OpenCV 구조 최적화

실제로 한 일:
- GPU warp/blend 확대
- 불필요한 `copy()` 축소
- overlap ROI만 계산
- 대시보드와 panorama 출력 분리
- headless benchmark 추가

목적:
- “지금 구조 안에서 어디까지 갈 수 있는지”를 먼저 확인

결과:
- 성능은 일부 올라갔지만
- headless benchmark에서도 stitched fps 상한이 대략 20fps대 초반에 머묾

결론:
- UI만 없앤다고 30/60fps로 가는 문제가 아니었음
- 병목은 화면 출력보다 엔진 구조 쪽에 더 있었음

### 2단계. FFmpeg direct 입력 실험

의도:
- OpenCV가 내부적으로 FFmpeg를 감싸는 구조 대신
- 우리가 FFmpeg를 직접 제어하면 입력 병목이 개선되는지 확인

실제로 추가한 것:
- `ffmpeg_runtime.py`
- `ffmpeg_reader.py`
- `--input-runtime opencv / ffmpeg-cpu / ffmpeg-cuda`

예상:
- `ffmpeg-cuda`가 붙으면 더 빨라질 수 있을 것

실제 결과:
- 오히려 `opencv + headless`보다 `ffmpeg-cuda + headless`가 느린 구간이 나옴

### 왜 FFmpeg direct가 더 느려졌나

핵심 이유는 이거다.

현재 `ffmpeg-cuda` 경로도 결국:

```text
RTSP/H.264
-> ffmpeg decode
-> rawvideo bgr24 stdout
-> Python bytes
-> numpy.ndarray
-> Python buffer/sync
-> GpuMat.upload()
-> stitch
```

즉, decode를 FFmpeg가 하더라도 결과를 `rawvideo bgr24`로 Python에 밀어넣고 있었기 때문에:

- Python bytes 생성
- `np.ndarray` 생성
- Python 버퍼링
- 다시 GPU upload

이 남아 있었다.

그래서 결론은:

> FFmpeg를 직접 쓴다는 사실 자체가 중요한 게 아니라,  
> full-rate 프레임이 Python-visible object가 되는 구조를 벗어났는지가 중요하다.

## 핵심 개념: Python-visible frame 경계

지금까지 논의에서 가장 중요한 결론은 다음이다.

### 문제의 본질

- `cv2.warpPerspective()`가 Python이라 느린 게 아니다
- `numpy.ndarray` 자체가 나쁜 것도 아니다
- 문제는 **full-rate 프레임이 Python 객체가 되고, Python이 그 프레임을 소유하는 구조**다

구체적으로는:
- Python이 프레임 버퍼를 들고 있음
- Python이 sync/pairing을 수행함
- Python과 native/GPU 사이 경계가 계속 생김
- GPU로 보내기 전 host frame이 반드시 만들어짐

이 구조를 유지하면:
- reader를 OpenCV로 하든
- FFmpeg로 바꾸든
- 일부 GPU를 더 쓰든

결국 상한선이 빨리 보이게 된다.

## 그래서 어떤 구조가 필요하다고 판단했나

최종적으로 필요한 구조는 다음이다.

```text
입력/디코드 -> pair/sync -> stitch -> encode -> output stream
```

그리고 이 full-rate media path는 **native runtime**이 소유해야 한다.

Python은 여기서 빠진다. 대신 Python은:
- start/stop
- config
- metrics/log
- manual calibration command
- operator workflow

만 담당한다.

즉:

```text
Python = control plane
Native runtime = data plane / media plane
```

## A / B / C 선택지

논의 과정에서 3가지 선택지를 정리했다.

### A. Standalone native runtime

예:
- `stitch_runtime.exe`

특징:
- 별도 엔진 프로그램
- Python은 control/UI
- 구현 시작이 가장 현실적

장점:
- 지금 Python 자산(UI/로그/수동 포인트)을 살리기 쉬움
- 위험이 가장 낮음
- 먼저 성능 측정하기 좋음

단점:
- 별도 runtime wrapper가 필요함
- 나중에 제품형 파이프라인으로 가면 wrapper를 더 세분화할 수 있음

### B. GStreamer plugin/runtime

특징:
- 미디어 파이프라인 안에 stitch block을 넣는 방식

장점:
- 실시간 파이프라인 제품형 구조에 강함
- 입력/디코드/스티칭/인코드/송출을 한 줄로 구성하기 좋음
- 장기적으로 가장 “제품형” 구조가 될 가능성이 큼

단점:
- 개발 난이도 높음
- Windows보다 Linux에서 훨씬 유리
- 지금 당장 시작하기엔 공사 범위가 큼

### C. FFmpeg 중심 runtime/filter

특징:
- decode/encode/streaming 중심을 FFmpeg/libav 계열이 잡는 방식

장점:
- 코덱/프로토콜/송출 제어에 강함
- 입력/출력 중심 시스템에는 매우 적합

단점:
- API 레벨이 낮고 복잡함
- 커스텀 stitch logic을 중간에 끼워 넣는 작업 난이도가 높음

## 지금 왜 A를 먼저 택하나

현재 기준으로는 `A`가 가장 현실적이다.

이유:
1. 기존 Python UI를 버리지 않아도 됨
2. 먼저 native hot path를 검증할 수 있음
3. 실패해도 공사 범위가 B/C보다 작음
4. 이후 B/C로 갈 수 있도록 코어를 재사용하는 방향으로 설계 가능

즉:
- 지금은 `A`로 시작
- 필요하면 이후 `B` 또는 `C` wrapper를 붙이는 전략

## 중요한 정정: A를 한다고 해서 무조건 프로세스 분리부터 시작하는 건 아님

의사결정 중간에 이 논의가 있었다.

### 잘못 이해하면 안 되는 점

`A`의 본질은 “Python이 프레임 hot path를 소유하지 않게 만드는 것”이지,  
첫날부터 무조건 `exe` IPC 구조를 완성하라는 뜻은 아니다.

가능한 구현 순서:
1. 먼저 C++ 코어 엔진 작성
2. 초기에는 Python binding으로 붙여도 됨
3. 단, Python이 `left_np/right_np`를 full-rate로 계속 들고 있으면 안 됨
4. 이후 운영형 구조가 필요하면 별도 프로세스로 승격

즉:
- 구현 순서 면에서는 `C++ 코어 -> binding -> 필요 시 process split`
가 현실적이다
- 하지만 구조 원칙은 여전히
  - Python은 control
  - native는 frame owner
여야 한다

## 최종 결과를 어떻게 볼 것인가

이 부분도 별도 의사결정이 있었다.

### 잘못된 방식
- Python이 full-rate stitched raw frame을 받아서 화면에 그림

문제:
- Python이 다시 hot path frame owner가 됨
- viewer/UI backpressure가 메인 엔진 fps에 영향을 줄 수 있음

### 선택한 방식
- **최종 encoded output stream 자체를 Python이 외부 클라이언트처럼 본다**

즉 구조는:

```text
native runtime:
  input/decode -> stitch -> encode -> final output stream

Python UI:
  control/metrics/log
  + final stream viewer launch
```

이 방식의 장점:
1. Python이 raw panorama frame을 소유하지 않음
2. operator가 실제 전송 결과를 그대로 볼 수 있음
3. viewer가 느려도 메인 엔진 hot path를 막지 않음

## low-rate debug snapshot은 왜 옵션인가

이 논의도 있었다.

결론:
- `final stream viewer`가 있으면 v1에선 충분하다
- `low-rate debug snapshot`은 seam overlay, calibration helper, internal diagnostics가 필요할 때만 추가

즉 우선순위:
1. final output stream viewer
2. optional low-rate snapshot

## 한 줄 결론

현재 프로젝트는:
- Python/OpenCV 프로토타입으로 충분히 많은 실험을 했고
- 병목이 단순 함수 문제가 아니라 **Python-visible full-rate frame 구조**임을 확인했다

따라서 다음 단계는:

> **Python은 control/UI로 남기고, 실제 media pipeline은 native runtime으로 옮기는 것**

이며,

> **operator preview는 Python raw frame viewer가 아니라 final output stream observer 방식으로 간다**

가 현재까지의 정리된 결론이다.
